# ISS-WF-CI-OVERRIDE-1 — Operational CI override agent (PRSteward) + counter-limited policy

| Severity | Module | Status | Workflow | Date |
|---|---|---|---|---|
| blocker | workflow/ci-policy | open | wf-20260703-impl-policy-071 | 2026-07-03 |

## Problem

Every workflow that lands a PR is blocked by pre-existing CI failures
(`ci/__vite_ssr_exportName__`, `storybook/PARSE_ERROR`, `biome cognitive-complexity`)
that originate on `main` HEAD and are owned by separate issues with
queued follow-up workflows. The §6.2 safety gate #2 forces the
Orchestrator to stop and ask the user on every such PR, even when the
PR's own diff has no overlap with the failure trace. This converts
trivial PR-matters into interactive Q&A.

## Goal

Define a narrow, auditable policy and a dedicated agent (the
**PRSteward**) that — for each failing CI check independently —
verifies three preconditions:

1. The failing log mentions **no file** in the PR's diff.
2. The failure class (sha1 of canonical error block) is registered in
   `.copilot/meta/ci-override-counters.json` with a non-null
   `owned_by_issue` row that points to a `wf-...` directory under
   `.copilot/tasks/{active,queued}/`.
3. The consecutive override counter is below `_limit` (default 5).

If all three hold, the PRSteward records a full audit trail
(`handoff.yaml.gate_results`, squash trailer, registry row, counter
file, PR description) and authorizes the merge. Otherwise it stops,
writes `NEEDS_REVIEW.md`, and sets `workflow_status: needs-review`.

## Honesty disclosures

- Pre-existing CI failures on main HEAD:
  - `ci/__vite_ssr_exportName__` → owned by ISS-TEST-WEB-001 (open, queued: wf-20260703-fix-066-vitest-bump) → **eligible** for override.
  - `storybook/PARSE_ERROR Unexpected JSX expression` on `apps/web-next/src/kit/*.tsx` (Badge, Card, Dialog, Wizard) and `apps/web-next/src/blocks/workspace/AsyncSelect.tsx` → **NOT** in the counter file, no owned issue with queued follow-up. The PRSteward's first encounter registers a new blocker issue per §6.3.
  - `biome cognitive-complexity in apps/api/src/modules/workspace/tg-{broadcasts,segments}.service.ts` → marked resolved-as-filed (ISS-CI-003, won’t fix as policy waiver) → NOT in the counter file as an active failing class.

- The "biome noise is policy, not a quality gate" framing from ISS-CI-003 is an **operational waiver**, not a code fix. It is NOT in the counter file because the `ci` job is currently green against biome. If biome regresses again, it will be a fresh failure class and PRSteward will register it.

- PR #93 (the ISS-WF-REG-002 fix) does **not** touch any file referenced by `storybook/PARSE_ERROR`. The override policy says the PRSteward can override `ci` (eligible, count 3 → 4) but **must escalate** on `storybook` (new class, no queued fix). Net decision: needs-review. The proposed follow-up is to split the registry-state fix off `fix/ISS-WF-REG-002-registry-state-drift` into a docs-only PR — that PR would have only the `ci` failure, which IS eligible, and would land under override.
