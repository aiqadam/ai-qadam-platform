## What

Fixes the `storybook` CI failure by injecting `@vitejs/plugin-react` into `apps/storybook`'s Vite config. The plugin transpiles JSX in `.tsx` files via Babel before rolldown 1.1.3's parser sees them, sidestepping rolldown's `JSX syntax is disabled` quirk.

## Why

`apps/storybook` declared `@storybook/react-vite@8.6.18` as its framework, which is a thin pass-through adapter that does NOT auto-inject any Vite plugin for JSX. Vite 8.1.0 (resolved transitively via `@astrojs/react` -> `vite@^8`) uses rolldown 1.1.3 as the default production bundler. rolldown's parser disables JSX by default for `.tsx` files, so every storybook build hit `PARSE_ERROR: Unexpected JSX expression` against 12 atoms/blocks in `apps/web-next/src/kit/*` and `apps/blocks/workspace/AsyncSelect.tsx`.

Astro's own build (`pnpm --filter @aiqadam/web-next build`) succeeds with the same rolldown because Astro configures JSX handling internally; Storybook 8 does not, so we wire `@vitejs/plugin-react` explicitly. This is the canonical Storybook 8 + Vite + React integration pattern.

## How

- **`apps/storybook/package.json`**: added `@vitejs/plugin-react@^5.2.0` as a direct devDep. Already in the workspace tree transitively; declaring it as direct devDep documents intent and protects against future pnpm hoisting changes.
- **`apps/storybook/.storybook/main.ts`**: imported `react` from `@vitejs/plugin-react` and prepended `react({ jsxRuntime: 'automatic' })` as the first plugin in `viteFinal`. The plugin runs Babel + `@babel/plugin-transform-react-jsx` on every `.tsx` file so rolldown never sees JSX.

## Risks

- Storybook peer-dep warning: `@storybook/react-vite@8.6.18` declares `vite ^4 || ^5 || ^6` in peerDependencies, but the resolved vite is `8.1.0` (transitively via `@astrojs/react`). This peer skew is **pre-existing** on `origin/main` and not introduced by this workflow.
